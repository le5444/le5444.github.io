# Agent Workbench Calibration - 2026-06-18

这份文件只解决一个问题：后续开发不要再沿着旧的“写作前端 / 功能堆叠 / 控制台大杂烩”惯性继续走。

## 1. 最新边界

- 公开入口、浏览器标题、README 首屏继续保持：`织梦写作台 / Zhimeng Writing Agent`。
- `灵枢 LumenOS` 只作为底层 Agent OS / Agent IDE 运行层，用来组织线程、记忆、Skills、工具、Provider、Gateway、Worker、审批、自动化和多工作区。
- 不再把公开产品改成 LumenOS-first，也不再写“织梦只是内置 Writing Agent”。
- 写作是重要内置能力，但不是整个产品边界。

## 2. 产品北极星

目标不是做一个小说写作前端，而是做一个中文 AI Agent 工作台：

```text
左侧：线程 / 项目 / 工作区导航
中间：AI 对话主线程 + 任务执行轨迹
右侧：上下文 / 文件 / Diff / 审批 / 状态
底部：终端 / 日志 / 问题 / Worker / Gateway，需要时出现
设置：模型、Provider、API、权限、Bridge，不抢默认首页
```

默认首页的第一主角必须是“用户给 AI 发任务，并看到 AI 如何读上下文、调用工具、产生 Diff、等待审批、继续推理”。其他能力只能服务这条主线。

## 3. 已吸收资料的可执行结论

来自两份 docx、项目上下文、`ai可借鉴源码` 目录和 `frontend(1).zip` 的共同结论：

- Codex / Claude Code / Qwen Code / Trae 这类 Agent 的核心不是聊天框，而是 `任务线程 + 上下文组装 + 工具调用 + 权限门 + 执行轨迹`。
- UI 应该是清晰工作台，不是能力矩阵 dashboard。
- 项目模式必须围绕本地目录：绑定目录、扫描目录、读文件、生成 Diff、审批写入、终端命令、运行日志。
- 对话模式不应强制绑定目录，但同样要支持附件、文件摘要、图片 / 多模态和会话存储。
- Provider/API 配置要像设置中心或配置工具，不能默认占右侧栏或全屏打断。
- 参考源码的价值在架构拆解：query loop、tool pool、permission gate、context pipeline、subagent、skills、hooks、MCP、provider proxy。落地时必须按本项目结构重写，不机械搬运。

## 4. 当前实现判断

当前项目已经不只是静态前端，已有这些真实链路：

- AI 对话发送、附件、图片 / 多模态 parts、会话消息存储。
- `Agent Loop` 初版：plan / context_pack / model / bridge request / Gateway result / follow-up reasoning。
- Gateway 工具入口、只读自动提交、写入审批门、`write_file` Diff 草案、审批队列。
- 项目文件扫描、索引、预览、挂入线程、生成 Diff 草案。
- Agent 线程、上下文附件、审批关联、运行日志、模型 Worker 预检、Memory Manager、Provider 中心。

但这些仍然不等于“完成”：

- 默认首页还要继续减噪，避免看起来像把所有后台能力堆到一起。
- 项目模式还缺统一任务流：目录树、文件、Diff、审批、终端、日志应该在一次任务里自然串起来。
- Provider/API 配置还有“能保存配置”和“真能拿模型列表、测试对话、回写状态”的产品闭环要继续打磨。
- Agent Loop 还没有成为稳定自动执行器，审批后续跑、多文件补丁、失败恢复、Worker/Gateway 事件订阅都需要继续补。

## 5. 止损规则

1. 不再为了“看起来强”增加新按钮、新徽章、新面板。
2. 不再把写作页、Skills、Memory、Provider、Worker 同时塞进默认首页抢主线。
3. 不再把 API 配置放回默认右侧栏；它属于模型中心 / 设置 / 桌面配置工具。
4. 不再把 TypeScript 设计稿当成已落地功能；必须有真实调用链和验证证据。
5. 不再让模型直接写文件；所有写入都进入 Diff / 审批 / Gateway 证据链。
6. 不再沿用旧的 LumenOS-first 公开边界。

## 6. 接下来最优先

### P0：理清默认首页

- 左侧只保留线程、项目、工作区导航。
- 中间只强调 AI 对话主线程、任务状态、输入框、附件。
- 右侧只显示上下文、文件、Diff、审批、状态。
- 高级能力进入设置、模型中心、能力库、底部 Panel 或二级入口。

### P1：做扎实 AI 对话

- 模型配置可保存、可激活、可获取模型列表、可测试对话。
- 输入框支持文本、文件、图片、多模态附件，并明确哪些内容进入模型请求。
- 模型不可用时不打断首页，只给轻量状态和修复入口。

### P2：做顺项目模式

- 绑定目录 -> 扫描目录 -> 选择文件 -> 预览 -> 挂入上下文 -> 生成 Diff -> 审批写入 -> 写后复核。
- 终端命令只走 allowlist / Gateway / 审批，不开放任意 shell。

### P3：推进 Agent Loop

- 把 `sendChat` 周围的 UI 编排继续下沉到 `src/os/kernel/agent-loop.ts`。
- 让工具结果、审批结果、Worker 事件和模型续推理成为同一条任务协议。
- 多文件 Diff 和审批后续跑优先于新概念。

## 7. 下一个小改动建议

先不要重构大文件。下一步最适合做：

```text
把默认首页右侧“文件”面板整理成步骤式项目文件工作流：
1. 绑定目录
2. 扫描目录
3. 选中文件
4. 预览文件
5. 挂入上下文
6. 生成 Diff 草案
7. 去审批
```

这一步直接服务 Codex 式项目任务链路，也能减少“点进去不知道东南西北”的感觉。
