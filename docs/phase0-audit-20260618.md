# Phase 0 Audit - 2026-06-18

这份审计只做一件事：基于当前代码，确认哪些链路是真正在跑的，哪些仍然是设计稿或半成品，并给出后续必须遵守的止损边界。

后续开发的产品北极星见 `docs/agent-workbench-calibration-20260618.md`：默认首页必须围绕 AI 对话主线程、项目文件、Diff、审批和运行状态收口，不能再回到写作前端或控制台堆叠。

## 1. 现在真正在跑的东西

- 公开边界已重新收口：`index.html` 和 `README.md` 首屏都保持“织梦写作台 / Zhimeng Writing Agent”，并把定位从单纯小说前端改为“中文写作与个人任务的 AI Agent 工作台”；`灵枢 LumenOS` 只作为底层 Agent OS / Agent IDE 运行层，不再作为公开产品第一标题。
- `AIChatPanel.tsx` 已有真实聊天发送链路：`sendChat(...)` 会发出请求、接收流式回复、支持停止生成，并把结果写回消息流。
- 聊天输入已接上上下文组装：当前任务会先经过 `planAgentIntent`、`selectAgentSkills`、`planPersonalOS`、`buildToolRouteBundle`、`buildExecutorBridgeManifest`，再进入 `sendChat`。
- 聊天结果会解析执行桥请求：响应文本里抽到的 bridge request 会进入本地请求队列。
- Agent Loop 已有首个安全执行入口：模型回复里产生的只读 bridge request 可自动提交到 Gateway，并把结果回填为运行快照；写入、命令、远程探针、Skill run 等高风险动作仍只排队，不会自动执行。
- 自动提交回填已进入主线程：只读工具结果会追加到聊天消息流和 AI 结果列表，便于继续推理和审计。
- 工具结果续推理已接入一轮：自动提交的只读工具结果会再喂给模型生成一次续答；该续答明确要求不要重复输出工具请求，避免无限循环。
- 单轮工具续推理协议已抽到 `src/os/kernel/agent-loop-bridge.ts`；`AIChatPanel` 只保留 UI 编排和状态回写，后续可继续迁移到 `src/os/kernel/agent-loop.ts`，避免把循环逻辑堆进聊天组件。
- `runAgentLoop` 已复用同一个工具结果回灌 prompt，UI 单轮续答和内核多轮循环不再各写一套工具结果提示词。
- AgentRun 完成记录现在优先使用工具续答作为最终摘要，记忆/KAIROS 观察不再只记录第一轮模型回复。
- `<bridge-request>` / 完成标记的展示清洗也已移入 `agent-loop-bridge.ts`，聊天组件不再维护协议清洗正则。
- `runAgentLoop` 现在可回调模型回复和循环提示，后续 UI 可以订阅内核事件而不是重复做整套轮询/展示协议。
- `AgentControlCenter.tsx` 已订阅 `runAgentLoop` 的 `onModelMessage` 和 `onLoopPrompt`，Agent Loop 的模型回复、工具结果续推理提示会进入线程消息和运行日志。
- `runAgentLoop` 已开始返回结构化执行结果：包含停止原因、最后阶段、工具结果列表和待审批项；工作台会把“任务完成 / 等待审批 / 工具异常 / 达到轮数上限”等状态回灌到右侧运行状态、线程事件和运行日志。
- Agent Loop 因审批停住时，当前线程会保存可续跑快照；审批执行或拒绝后，运行状态面板会出现“继续”入口，把审批结果注入原任务并重新进入同一 Agent Loop 链路。
- 运行报告已拆出 `## Agent Loop 轨迹`，不再把 Agent Loop 证据混在普通日志里。
- 对话区已支持附件、多模态、文件片段、运行摘要和折叠高级工具；直接对话会把图片、解析文件、元数据和失败附件区分记录。
- 直接对话的附件传输已进入运行日志：`对话附件已进入模型请求` 会明确标出图片作为 vision/multimodal parts，文件解析内容作为文本上下文，避免把“选择了附件”误判成“模型已读懂附件”。
- 运行报告已拆出 `## 直接对话 / 附件传输`，方便审计一次聊天里哪些内容真的进入了模型请求。
- 模型未连接时的主页反馈已收敛：发送内容只保存用户消息、线程事件和运行日志，不再自动打开模型中心，也不再在主聊天区追加“等待模型连接”的助手错误卡；输入框附近保留轻量状态与重试/模型设置入口。
- 中间主线程已加入轻量任务状态条：把当前任务、Agent Loop 状态、上下文数量、审批数量、变更数量和执行/续跑入口收在消息流上方，让默认首页更接近 Codex 式任务工作台，而不是纯聊天气泡列表。
- 主线程里的工具/系统消息已进一步压缩为任务事件流样式：默认显示一行工具摘要和状态，展开后再看细节，减少工具日志卡片打断对话主线。
- 右侧 Secondary Sidebar 的窄图标栏已增强可读性：每个入口的 title/aria 包含用途和数量，非零计数才显示徽标，当前入口保留更清晰的 active 指示，继续保持默认不展开大面板。
- 左侧线程列表已进一步扫读优化：线程行 meta 改成轻量状态芯片，显示对话模式、更新时间、非零审批/变更/上下文数量，让线程列表更像 Codex 的任务列表，而不是普通文件夹清单。
- 项目模式的目录绑定入口已更清晰：没有本地目录时显示“绑定目录”，绑定后才显示“扫描”，避免用户在未绑定项目时误以为项目模式已经可用。
- `workspace_scan` 成功后会优先把目录索引关联到匹配的项目线程，并写入 `workspace_scan 已关联项目线程` 日志，减少扫描结果落到自由对话线程的风险。
- 项目文件链路已经从“能扫描”推进到“能读上下文”：绑定项目目录后可扫描索引、选择索引文件、读取预览，并把 `read_file` 结果作为结构化 `thread_context` 附到当前线程。
- 右侧文件预览已经能生成 Diff 草案：Home 文件面板和完整工作台侧栏都可基于当前 `read_file` 预览创建待审查 hunk，打开右侧 Diff 面板，支持先接受/拒绝 hunk，再生成写入审批。
- 直接对话里的 `<bridge-request action="write_file">` 不再直接写文件；前端会把它转成待审查 Diff 草案，并继续走 Gateway / 审批链路，避免模型回复绕过人工确认。
- 右侧 Diff 面板已有“接受全部 / 拒绝全部 / 生成写入审批 / 清空”动作，并明确提示“先接受 hunk，这一步不会写文件”。
- `write_file` 审批执行结果已开始回灌到前端状态：Gateway `approval_decide` 返回成功后，已接受的 Diff hunk 会标记为完成，并追加线程事件与运行日志；拒绝或失败时，未完成 hunk 会标记为拒绝。
- `write_file` 审批执行成功后已接入自动 `read_file` 复核：如果 Gateway 在线且读闸门开启，会读取写入后的文件片段，写入运行日志、工具消息，并作为 `thread_context` 文件附件挂回当前线程；如果读闸门未开，会明确记录跳过原因。
- 右侧上下文面板可把当前文件预览、活跃 Skills、候选 Skills、选中的 Memory 记录附加到线程，也能跳转到 Skills / Memory 管理视图。
- 右侧运行状态面板已增加 Agent Loop 速览：显示任务来源、规划/上下文/工具请求/Gateway/结果回灌链条，并可从当前线程触发一次 Agent Loop。
- 右侧运行状态面板已增加审批、Worker、终端快捷入口：可查看当前线程审批队列、最近 Worker 任务和白名单命令快捷运行；终端命令仍受 Gateway 权限和 allowlist 约束。
- 右侧辅助栏已做一轮视觉减负：在 `codex-workbench-full` 浅色工作台下收窄 Secondary Sidebar，降低卡片背景和边框重量，压缩状态/审批/Worker/终端快捷块的间距和按钮高度，让默认页面更像 IDE 辅助面板而不是调试仪表盘。

## 2. 现在还没真正跑通的东西

- Agent Loop 还没有完整闭环：目前已从“组装上下文 -> `sendChat` -> 解析桥请求”推进到“安全只读请求自动提交 -> 工具结果回填 -> 多轮循环状态回灌 -> 审批结果手动续跑”，但还没有稳定的自动中断恢复、审批后无感续跑和多文件任务收束。
- 多数高级能力仍依赖手动点击按钮，而不是从一次任务里自然路由和自动推进。
- 项目模式的完整闭环还不完整：虽然已有绑定/扫描/目录索引关联、文件预览、Diff 草案、审批生成、终端快捷运行、单次写入审批结果回灌和写后 `read_file` 复核，但目录监听、失败恢复、多文件补丁任务流和真实文件树持续同步还没有形成统一产品闭环。
- 图片/多模态目前已能作为模型请求 parts 传输和记录，但最终识别质量仍取决于已选 Provider/模型是否真的支持 vision。
- 会话存储、线程报告和运行日志已有可用基础，但还不是 Codex/Claude Code 那种从任务输入到工具执行再到补丁/审批的完整工作流。
- 右侧/底部的很多能力面板虽然在收口并能承载文件、Diff、审批、Worker、终端、Skills、Memory，也已做一轮浅色辅助栏减负，但信息架构还需要继续统一，离 Codex/Claude Code 式主线程工作台仍有距离。
- Agent Loop 已经可见、可触发、可记录事件，并能说明本轮为什么停止；审批决策后也能从运行状态手动续跑。但它还不是能稳定自动恢复任务、自动选择工具、连续修补文件并产出最终结果的自主执行器。
- Skills、Memory、Worker 已能被附加或查看，但主要仍是手动辅助上下文，还没有形成自动检索、自动注入、自动委派和自动回收结果的编排链路。
- exe 桌面形态、独立 Provider 配置工具、权限审批产品化还没有完成。

## 3. 设计稿 / 概念层

- `docs/Personal-OS-Roadmap.md` 里很多阶段能力已经进入详细路线图，但其中相当一部分仍是未来目标，不应被误当成当前已完成能力。
- `docs/项目路线图.md` 仍偏宏观规划，适合做方向参考，不适合当作当前实现完成度证据。
- `README.md` 仍在描述更广泛的工作台愿景，不等于所有能力都已落地。

## 4. 本轮止损清单

1. 不再继续加新的按钮、徽章、状态块，先收口现有入口。
2. 不再把首页做成调试仪表盘，默认首页必须保持 AI 对话为第一主角。
3. 不再把 Provider/API 配置塞回右侧主面板，继续保留设置/模型中心路径。
4. 不再把写作域当成产品边界，写作只能是内置 domain 之一。
5. 不再把“已写出 TS 设计稿”当成“功能已完成”；没有真实运行闭环的都只能算待落地。
6. 不再把文件写入做成模型直写；所有 `write_file` 都必须先进入 Diff / 审批 / Gateway 证据链。

## 5. 下一步优先级

1. 继续补 Agent Loop 的真实执行闭环，而不是只保留按钮触发。
2. 把项目模式的目录树、文件、Diff、终端、审批统一到同一任务流里。
3. 开始处理多文件补丁任务流：一次模型回复可以产生多个文件 Diff，各自审查、审批、复核。
4. 继续做 UI 信息架构收束：下一步应检查左侧线程/项目创建入口是否还过重，统一成“顶部新建 + 行级菜单”的简单路径。
5. 再往后才是 Skills、Memory、Worker、Runbook 的自动化联动。
