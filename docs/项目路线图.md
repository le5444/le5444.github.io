# 织梦写作台路线图

这份文档只回答一个问题：打开项目时，它到底应该是什么。

**织梦写作台 / Zhimeng Writing Agent** 是公开入口。它不是单纯的小说写作前端，而是老大的中文 AI Agent 工作台：默认围绕 AI 对话、项目、文件、上下文、工具、审批、记忆和 Skills 完成任务。写作是其中最重要的内置 Agent / domain 之一，但不是产品天花板。

底层可以继续演进成更强的 Agent OS / Agent IDE 运行层，但公开界面和 README 不再把抽象系统名放在第一层。用户点进来，应该先知道这是“我的项目工作台”，能直接对话、配置模型、绑定项目、读写文件、审查 Diff、执行受控工具并继续推进任务。

## 目标分层

### 3 个大目标

1. **Agent Home 像 Codex / Claude Code / VS Code**
   默认首页必须是左侧线程 / 项目导航、中间 AI 对话主线程、右侧上下文 / 文件 / Diff / 审批 / 状态，底部终端按需出现。不能退回杂乱 dashboard，也不能让 API 配置默认占右侧或全屏打断。

2. **核心 Agent 链路真实跑通**
   从用户输入、附件和图片开始，经过线程保存、上下文打包、Provider API、模型回复、`<bridge-request>` 工具请求、Gateway / Diff / 审批、工具结果回灌，再继续推理或明确完成。不能只停在 `sendChat`，也不能把 TypeScript 设计稿当成真实能力。

3. **桌面产品可用而且可维护**
   最终面向 exe 桌面应用：模型配置工具、Provider 切换、项目目录绑定、文件权限、命令审批、运行报告、错误恢复、密钥边界、构建发布和审美打磨都要形成稳定闭环。

### 5 个中目标

1. **AI 对话与模型配置**
   自定义 baseURL / API key / 模型 ID 能保存启用，能获取账号 `/models`，能真实发送文本、附件和图片，多模态内容进入 Provider 请求；失败时保存线程并给模型设置和重试入口。

2. **Agent Home UI 精简**
   左侧只放线程、项目、工作区导航；中间只突出对话、任务状态和输入；右侧只放上下文、文件、Diff、审批、状态；复杂功能收进菜单、右侧标签或底部面板。

3. **项目模式闭环**
   对话模式不绑定目录；项目模式绑定本地文件夹，支持目录扫描、文件读取、上下文挂载、写文件 Diff 草案、审批、终端命令和运行报告。

4. **Agent Runtime 能力层**
   Agent Loop、Runbook、Instruction Stack、Memory、Skills、Worker、Subagent 逐步接到同一条可审查链路里，做到能计划、能调用工具、能回灌、能续跑。

5. **桌面与稳定性**
   Provider 配置工具、Bridge / Gateway doctor、桌面打包、公开入口、缓存退役、错误提示、空状态、白色轻量设置浮层和审美统一都要进验收门禁。

### 当前 1 个小目标

**目标锁定与进度门禁**：把上面的 3 大目标、5 中目标和当前小目标写进项目路线图，并让自动验证检查这些关键词。这样以后继续开发时，任何线程都能从项目文件里读到方向，不会又沿着旧的“写作前端 / 功能堆叠”惯性跑偏。

## 当前产品形状

```text
左侧栏      对话线程、项目、工作区入口
中间主区    AI 对话、消息流、输入框、附件
右侧栏      上下文、文件 / Diff、审批、终端 / 日志
设置        模型接口、API key、Provider、权限和本地 Bridge
```

设计参考 VS Code / Codex / Claude Code 的工作台结构，但默认用中文命名，不堆一屏看不懂的按钮。

## 阶段目标

### Phase 0：先止乱

- 公开入口统一为“织梦写作台 / Zhimeng Writing Agent”。
- 首页默认突出 AI 对话和项目工作区，不再像杂乱 dashboard。
- API 配置进入设置或专门配置入口，不默认占据主界面。
- 把旧术语、重复按钮、难退出的全屏面板逐步清掉。

### Phase 1：把 AI 对话做扎实

- 模型配置可保存、切换、测试。
- 发送消息能稳定拿到回复。
- 支持文本文件、代码文件、Markdown、JSON、图片等附件。
- 图片和多模态内容先进入消息与上下文，后续接入支持视觉的模型。
- 会话能保存、恢复、关联附件和上下文。

### Phase 2：把项目模式做清楚

- 对话模式：不绑定固定目录，随时聊天和传文件。
- 项目模式：绑定本地文件夹，显示目录树，保留项目会话。
- 文件读取、修改草案、Diff、审批流程形成闭环。
- 终端命令、目录扫描、文件监听都放在明确的项目边界里。

### Phase 3：把工具链接通

- 文件读写、命令执行、网页获取、模型探针都走本地 Bridge / Gateway。
- 高风险动作先生成草案和审批，不直接改文件。
- Diff、审批、运行日志和上下文附件能在同一条线程里追溯。

### Phase 4：把 Agent 工作流跑起来

- 让任务从“用户输入”进入计划、上下文、模型回复、工具请求、工具结果、继续推理的循环。
- Skills、记忆、项目规则、Runbook 进入可审查上下文。
- Worker 和后台任务只做明确、可追踪、可取消的事情。

### Phase 5：桌面应用化

- 面向 exe 桌面应用打磨启动、配置、权限和本地文件访问。
- 做一个更顺手的模型配置工具，不要求每次在网页里填 API。
- 稳定性、审美、空状态、错误提示和新手路径继续打磨。

## 执行原则

- 先把主路径做清楚，再加高级功能。
- 默认界面少而准，复杂能力收进设置、右侧栏或底部面板。
- 能中文说清楚的地方就用中文。
- 不为了展示架构而展示架构，用户看到的应该是任务、文件、对话、审批和结果。
