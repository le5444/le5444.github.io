# 织梦写作台路线图

这份文档只回答一个问题：打开项目时，它到底应该是什么。

**织梦写作台 / Zhimeng Writing Agent** 是公开入口。它首先是一个面向中文长篇小说创作的 AI 工作台：能聊天、能看项目文件、能整理上下文、能调用写作 Skills、能生成修改草案，并把高风险动作放进审批。

底层可以继续演进成更强的个人 AI 工作流，但公开界面和 README 不再把抽象系统名放在第一层。用户点进来，应该先知道这是“我的写作项目工作台”。

## 当前产品形状

```text
左侧栏      对话线程、项目、工作区入口
中间主区    AI 对话、消息流、输入框、附件
右侧栏      上下文、文件 / Diff、审批、终端 / 日志
设置        模型接口、API key、Provider、权限和本地 Bridge
```

设计参考 VS Code / Codex / Claude Code 的工作台结构，但默认用中文命名，不堆一屏看不懂的按钮。

## 阶段目标

### Phase 0：先止乱

- 公开入口统一为“织梦写作台 / Zhimeng Writing Agent”。
- 首页默认突出 AI 对话和项目工作区，不再像杂乱 dashboard。
- API 配置进入设置或专门配置入口，不默认占据主界面。
- 把旧术语、重复按钮、难退出的全屏面板逐步清掉。

### Phase 1：把 AI 对话做扎实

- 模型配置可保存、切换、测试。
- 发送消息能稳定拿到回复。
- 支持文本文件、代码文件、Markdown、JSON、图片等附件。
- 图片和多模态内容先进入消息与上下文，后续接入支持视觉的模型。
- 会话能保存、恢复、关联附件和上下文。

### Phase 2：把项目模式做清楚

- 对话模式：不绑定固定目录，随时聊天和传文件。
- 项目模式：绑定本地文件夹，显示目录树，保留项目会话。
- 文件读取、修改草案、Diff、审批流程形成闭环。
- 终端命令、目录扫描、文件监听都放在明确的项目边界里。

### Phase 3：把工具链接通

- 文件读写、命令执行、网页获取、模型探针都走本地 Bridge / Gateway。
- 高风险动作先生成草案和审批，不直接改文件。
- Diff、审批、运行日志和上下文附件能在同一条线程里追溯。

### Phase 4：把 Agent 工作流跑起来

- 让任务从“用户输入”进入计划、上下文、模型回复、工具请求、工具结果、继续推理的循环。
- Skills、记忆、项目规则、Runbook 进入可审查上下文。
- Worker 和后台任务只做明确、可追踪、可取消的事情。

### Phase 5：桌面应用化

- 面向 exe 桌面应用打磨启动、配置、权限和本地文件访问。
- 做一个更顺手的模型配置工具，不要求每次在网页里填 API。
- 稳定性、审美、空状态、错误提示和新手路径继续打磨。

## 执行原则

- 先把主路径做清楚，再加高级功能。
- 默认界面少而准，复杂能力收进设置、右侧栏或底部面板。
- 能中文说清楚的地方就用中文。
- 不为了展示架构而展示架构，用户看到的应该是任务、文件、对话、审批和结果。
